# 제품 요구사항 문서 (PRD) — competitive-growth-monitor

**버전:** 0.3  
**작성일:** 2026-05-08  
**상태:** MVP 확정

---

## 1. 개요

`competitive-growth-monitor`는 **Competitor Action Monitoring** 시스템입니다.

경쟁사의 광고 활동, 매물/공급 규모, 정책·프로모션 변경을 함께 추적하여 *경쟁사가 무엇을 하려는지* 추론하고, 마케팅 및 공급 팀이 선제적으로 대응할 수 있도록 지원합니다.

단순히 광고 수를 세는 것이 목적이 아닙니다. 세 가지 핵심 시그널(광고, 공급, 정책)이 동시에 움직일 때 드러나는 패턴에서 경쟁사의 전략적 의도를 읽어냅니다.

수집된 원시 데이터는 Google Sheets에 누적되며, 팀은 원시 시트를 직접 보는 것이 아닌 **Looker Studio 대시보드**를 통해 인사이트를 확인합니다.

---

## 2. 문제 정의

현재 팀은 경쟁사 동향을 비정기적으로 수동 확인합니다. 이로 인해 경쟁사의 신시장 진입, 광고 전략 변화, 정책 개정과 같은 중요한 시그널이 늦게 포착되거나 놓치는 경우가 발생합니다. 이는 마케팅 캠페인 기획과 공급 전략 수립에 불필요한 지연을 초래합니다.

---

## 3. 목표

1. 경쟁사의 광고 활동, 공급/매물 규모, 정책·프로모션 변경 시그널을 자동으로 수집한다.
2. 수집 데이터를 Google Sheets에 원시 스냅샷 형태로 누적한다.
3. 팀원이 Looker Studio 대시보드를 통해 단일 시그널 트렌드와 **복합 액션 시그널**을 직관적으로 파악할 수 있게 한다.
4. 광고 급증, 매물 급증, 정책·프로모션 변경, 복합 시그널, 수집 실패 시 Google Chat으로 즉각 알림을 발송한다.
5. 인프라를 최소화하고 유지보수가 쉬운 구조를 유지한다.

---

## 4. 범위 외 (Non-Goals)

- 33m2 자사 데이터 수집 — 이 도구는 외부 경쟁사 전용입니다.
- 실시간 모니터링 — 일 1회 이상의 수집으로 충분합니다.
- 정밀 광고 지출 추정 — 표면적 시그널 수준에서 수집합니다.
- 원시 HTML 또는 전체 페이지 아카이빙 — 구조화된 필드와 해시만 저장합니다.
- 광고 지출 금액 추정 — Meta Ad Library와 Google Ads Transparency Center는 지출 데이터를 공개하지 않습니다.

## 6. 수집 시그널 정의

### 6.1 공급 시그널 (`raw_supply_snapshots`)
- 수집 방법: 경쟁사 공개 검색 결과 페이지 파싱 (로그인 불필요)
- 수집 데이터: 시장(도시)별 활성 매물 수
- 활용: 신시장 진입 감지, 급격한 성장 또는 수축 파악

### 6.2 광고 시그널 (`ad_activity_snapshots`)
- **수집 소스 (MVP):** 마케팅 팀이 현재 수동으로 사용 중인 공개 광고 라이브러리
  - Meta Ad Library: https://www.facebook.com/ads/library/
  - Google Ads Transparency Center: https://adstransparency.google.com/?region=KR
- 두 소스 모두 로그인 없이 접근 가능한 공개 페이지입니다. 이용약관을 준수합니다.
- MVP 수집 항목: 광고주명(advertiser_name), 광고 소재(creative_text), 광고 형식(creative_format), 랜딩 URL, 활성 광고 수(active_ad_count), 신규 광고 수(new_ad_count), 광고 상태(ad_status), 관측일(date), 소스 플랫폼(source_platform)
- **예상 지출(estimated_spend)은 수집하지 않습니다.** 두 소스 모두 광고 지출 금액을 공개하지 않습니다.
- 활용: 신규 캠페인 감지, 광고 소재 및 메시지 전략 변화, 계절성 광고 파악, 경쟁사 광고 활동 증감 추이

### 6.3 정책 시그널 (`policy_change_log`)
- **MVP 핵심 시그널입니다. 부가 기능이 아닙니다.**
- 수집 방법: 공개 정책 페이지 주기적 크롤링 (로그인 불필요), 텍스트 SHA-256 해시 비교
- 모니터링 페이지 유형: 이용약관, 호스트·공급자 약관, 취소/환불 정책, 프로모션·인센티브 페이지, 고객센터 FAQ
- 수집 데이터: 변경 감지 여부, 이전/현재 해시, 변경 내용 요약 excerpt
- 활용: 수수료 정책, 취소 정책, 호스트 약관, 프로모션 조건 변경 조기 감지. 광고·공급 시그널과 결합하면 경쟁사 전략 변화를 추론할 수 있습니다.

### 6.4 바이럴 평판 시그널 (`viral_reputation_mentions`)
- 수집 방법: 공개 SNS 검색, 뉴스, 커뮤니티 (Twitter/X, Reddit, 네이버 블로그 등)
- 수집 데이터: 언급 URL, 제목, 감성(sentiment), 참여도 지표
- 활용: 부정 이슈 조기 포착, 바이럴 캠페인 파악

### 6.5 알림 이력 (`alert_log`)
- 발송된 Google Chat 알림 전체를 기록합니다.
- 알림 유형: 정책 변경, 공급 임계값 초과, 평판 급등, Job 실패
- 팀의 후속 조치 확인 여부(`acknowledged`)를 수동으로 기록할 수 있습니다.

---

## 6.6 복합 액션 시그널 (Combined Action Signals)

개별 시그널이 아니라 여러 시그널의 동시 변화가 경쟁사의 의도를 드러냅니다. 대시보드의 "복합 액션 시그널" 뷰에서 팀이 수동으로 해석합니다.

| 관찰된 시그널 조합 | 추정 경쟁사 행동 |
|---|---|
| 광고 활동 증가 + 매물 수 증가 | 수요·공급 확장 캠페인 가능성 |
| 호스트 정책 변경 + 매물 수 증가 | 공급자 확보(acquisition) 드라이브 가능성 |
| 신규 랜딩 URL + 프로모션 페이지 변경 | 신규 캠페인 또는 오퍼 런칭 가능성 |
| 광고 메시지 변경 + 정책 변경 | 리포지셔닝 또는 가격 전략 변화 가능성 |

이 해석 레이어는 자동화하지 않습니다. 수집은 자동, 해석은 팀이 합니다.

## 7. 데이터 저장

- **Google Sheets**를 MVP 단계의 기본 저장소로 사용합니다.
- 행은 append-only입니다. 수정 또는 삭제 없이 누적합니다.
- 팀은 원시 시트를 직접 사용하지 않으며, Looker Studio 대시보드를 통해 접근합니다.
- 탭 구성: `raw_supply_snapshots`, `ad_activity_snapshots`, `policy_change_log`, `viral_reputation_mentions`, `alert_log`

## 8. 대시보드

Looker Studio를 Google Sheets 데이터 소스로 연결합니다. 대상 사용자: 마케팅 팀, 공급 팀 리드

대시보드는 원시 데이터를 그대로 보여주는 것이 아니라, 팀이 경쟁사 행동을 해석할 수 있도록 구성합니다.

| 섹션 | 설명 |
|---|---|
| **경쟁사 개요 (Competitor Overview)** | 경쟁사별 최근 시그널 요약 — 마지막 수집 시각, 활성 광고 수, 매물 수, 최근 정책 변경 여부 |
| **매물/공급 트렌드 (Supply Trend)** | 시장·경쟁사별 매물 수 시계열 차트 |
| **광고 활동 트렌드 (Ad Activity Trend)** | 활성 광고 수 변화, 신규 광고 감지, 광고 소재 및 랜딩 URL 타임라인 |
| **정책·프로모션 변경 로그 (Policy & Promotion Log)** | 감지된 정책 변경 이력, 변경 내용 요약, 경쟁사·페이지 필터 |
| **복합 액션 시그널 (Combined Action Signals)** | 광고·공급·정책 시그널이 겹치는 구간 강조 — 팀이 경쟁사 의도를 해석하는 뷰 |
| **알림 이력 (Alert History)** | 발송된 Google Chat 알림 전체, 확인 여부 추적 |

---

## 9. 알림

Google Chat Incoming Webhook을 통해 발송합니다. 알림은 중요한 시그널에만 발송합니다. 과도한 알림은 팀의 주의를 분산시킵니다.

| 알림 유형 | 트리거 조건 | 우선순위 |
|---|---|---|
| **정책·프로모션 변경** | 모니터링 대상 페이지 텍스트 변경 감지 | MVP |
| **수집 실패** | Job 실행 오류 발생 시 | MVP |
| **광고 활동 급증** | 활성 광고 수 직전 대비 일정 비율 이상 증가 | V2 |
| **매물/공급 급증** | 시장별 매물 수 주간 20% 이상 변동 | V2 |
| **복합 액션 시그널** | 동일 경쟁사에서 광고·공급·정책 시그널이 동시 변화 | V2 |

## 11. 권한 및 접근 정책

### Google Sheets
- 서비스 계정에 대상 스프레드시트 편집자 권한 필요
- 인증 정보는 `GOOGLE_APPLICATION_CREDENTIALS` 환경 변수로 관리

### Google Chat
- Incoming Webhook URL 생성 필요 (Google Chat space → Apps & integrations → Incoming Webhooks)
- `GOOGLE_CHAT_WEBHOOK_URL` 환경 변수로 관리
- 하드코딩 금지

### 광고 데이터 수집 소스
- Meta Ad Library (https://www.facebook.com/ads/library/) — 로그인 불필요; 이용약관 준수
- Google Ads Transparency Center (https://adstransparency.google.com/?region=KR) — 로그인 불필요; 이용약관 준수

### 공개 웹 크롤링
- 로그인이 필요 없는 공개 페이지만 대상
- 각 경쟁사 robots.txt 준수
- 적절한 요청 딜레이 유지 (기본 1–3초 간격)
- 과도한 트래픽 유발 금지

## 12. 성공 지표

- 4개 Job 모두 일 1회 성공률 95% 이상
- 출시 후 1주일 내 대시보드 정기 사용 시작
- 출시 후 30일 내 수집 데이터 기반 의사결정 사례 1건 이상

## 13. 리스크

| 리스크 | 완화 방안 |
|--------|----------|
| 경쟁사 사이트의 스크래퍼 차단 | User-Agent 변경, 재시도 backoff, 수동 대체 절차 문서화 |
| 광고 라이브러리 JS 렌더링 | 공식 Meta Ad Library API 또는 Google Ads Transparency API로 전환 |
| Google Sheets API 속도 제한 | 배치 쓰기, 지수 backoff 적용 |
| 정책 페이지 URL 변경 | `policy_pages.yaml` 분기별 검토, URL 헬스체크 추가 |
| 광고 라이브러리 UI 구조 변경 | 수집 로직 점검 및 수동 수집으로 임시 대체 |
